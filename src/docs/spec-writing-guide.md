# 기능 명세서 작성 가이드 (Spec Writing Guidelines) v1.2

## 1. 개요 및 기본 원칙
본 가이드는 기획/PM이 작성한 기능 명세서가 개발팀에게 명확하게 전달되도록 하는 것을 목적으로 합니다.

### 핵심 원칙
1.  **What, Not How**: '어떻게 구현(Code)했는가'가 아닌 **'무엇이 동작(Behavior)하는가'**를 기술합니다.
2.  **Explicit Logic**: 모호한 표현을 피하고, **구체적인 조건(Trigger)과 결과(Action)**를 명시합니다.
3.  **UI Interaction**: 사용자의 행동에 따른 시스템의 시각적/기능적 반응을 정의합니다.

---

## 2. 작성 세부 지침

### 2.1 UI/UX 요소 기술 방법
단순한 존재 여부가 아닌, **입력 방식**과 **피드백**을 구체화합니다.

* **입력 컴포넌트**: "검색 기능" (X) -> "검색 항목 **선택(Select)** 후 검색어 **입력(Input)**" (O)
* **상태 반영**: "필터 적용됨" (X) -> "하단에 **칩(Chip)** 형태로 조건 나열, 개별 삭제 가능" (O)

### 2.2 데이터 및 비즈니스 로직 기술 방법
개발자가 코드를 보지 않고도 구현 가능하도록 **규칙**을 설명합니다.

* **추상화(Abstraction)**: `MOCK_DATA` 변수명 (X) -> **"차량 정보 데이터베이스"** 또는 **"등록된 차량 정보"** (O)
* **계산 공식**: "경과일 노출" (X) -> "`현재 일시 - 생성 일시`를 계산하여 표기" (O)

### 2.3 리스트(Table) 기술 표준
1.  **컬럼 정의**: 헤더명, 정렬(Sort) 지원 여부, 데이터 포맷.
2.  **페이지네이션**: 페이지당 노출 건수 (예: 40건).
3.  **예외 처리**: 데이터 0건 시 "결과가 없습니다" 문구 노출 등.

### 2.4 동적 동작 및 의존성 (Dynamic Logic)
특정 필드의 값에 따라 다른 필드나 UI가 변하는 경우를 명시합니다.

* **의존적 필터**: "파트너 유형 선택 시, 진행 상태 필터의 옵션 목록이 해당 유형에 유효한 값으로 갱신됨."
* **초기화 로직**: "상위 필터(지역1) 변경 시, 하위 필터(지역2) 값은 초기화됨."

### 2.5 복합 레이아웃 구조 (Complex Layouts)
Drawer, Modal 내부에 탭(Tab) 등 구분자가 있는 경우 계층 구조로 기술합니다.

* **구조 명시**: [Drawer] > [Tab: 상세정보] > [Section: 기본 정보] 순으로 위계 표현.
* **진입점**: "리스트의 행(Row) 클릭 시 우측 Drawer가 열리며, 기본적으로 '상세정보' 탭이 활성화됨."

### 2.6 파생 데이터 및 집계 (Derived Data) **(New)**
DB에 저장된 원본 데이터가 아니라, 프론트엔드나 쿼리 시점에 계산되어야 하는 값의 로직을 정의합니다.

* **집계 로직**: "대상 차량수 = 해당 정책 ID와 매핑된 전체 차량 레코드 수"
* **비율 계산**: "진행률 = (수행 완료 상태인 차량 수 / 전체 대상 차량 수) * 100"

### 2.7 대량 입력 및 배치 처리 (Bulk Input) **(New)**
텍스트 영역 등을 통해 다수의 데이터를 한 번에 처리하는 로직을 명시합니다.

* **구분자 정의**: "사용자가 입력한 텍스트를 **콤마(,)** 또는 **줄바꿈(Enter)**으로 구분하여 배열로 변환."
* **유효성 검사**: "빈 값(Empty string)은 무시하고 유효한 데이터만 필터링하여 처리."

## 3. 지양해야 할 표현
| 지양해야 할 표현 (Bad) | 권장 표현 (Good) |
| :--- | :--- |
| `useState`로 관리 | 사용자가 입력하는 즉시 반영됨 |
| 더미 데이터(Mock) 노출 | (실제 운영될 데이터 규칙 기술) |
| 미션 등록 기능 (코드에 없음) | (실제 코드에 구현된 기능만 기술) |
